# Hotfix 文档 — 2026-04-13 文件上传与列表显示修复

## 概述

本次修复解决了 `./calcite-web` 项目中与文件管理相关的 **2 个前端 Bug**：

1. **Bug-1**：在笔记编辑页通过 `md-editor-v3` 上传图片/文件时，`note_id` 未能正确绑定到上传资源。
2. **Bug-2**：在笔记列表界面（以及编辑界面的文件面板），用户无法正确查看已上传文件列表，点击刷新也无效。

修复严格对照 `./docs/api.md` 第 716~982 行的 API 说明进行。

---

## Bug-1：上传文件未绑定 note_id

### 根因分析

- API 文档要求 `/api/file/upload` 使用 `multipart/form-data` 上传，支持附加字段 `note_id`（`int64`，可选）。
- 原实现 `calcite-web/src/api/file.js` 中的 `uploadFile` 函数**显式设置了请求头**：
  ```js
  headers: {
    'Content-Type': 'multipart/form-data'
  }
  ```
- **问题**：当 axios（v1.13.2，使用 XHR adapter）发送 `FormData` 时，如果手动指定 `Content-Type: multipart/form-data` 却**没有携带 `boundary`**，浏览器不会自动补充 boundary。后端（Drogon）在解析 multipart body 时无法正确识别字段边界，导致 `note_id` 字段丢失或解析失败，文件虽然能上传成功，但无法与当前笔记关联。
- 此外，`NoteEditor.vue` 中原代码手动在组件内创建 `FormData` 并 `append('note_id', ...)`，却未利用 `uploadFile(formData, noteId)` 的第二个参数，职责划分不清晰。
- **第一轮修复的偏差**：直接移除 `Content-Type` 头后，axios 实例默认的 `Content-Type: application/json` 被保留，导致请求以 JSON 类型发送 FormData body，后端完全无法解析，出现"没有上传文件"错误。因此需要将 `Content-Type` 设为 `null`，让 axios 跳过该 header，由浏览器自动补充正确的 `multipart/form-data; boundary=...`。

### 修复内容

#### 文件 1：`calcite-web/src/api/file.js`

将 `Content-Type` 设为 `null`，覆盖 axios 实例默认的 `application/json`，同时让浏览器在发送 `FormData` 时自动补充带 `boundary` 的 `multipart/form-data`。

```diff
  return request({
    url: '/file/upload',
    method: 'post',
-   data: formData,
-   headers: {
-     'Content-Type': 'multipart/form-data'
-   }
+   data: formData,
+   headers: {
+     // 将 Content-Type 设为 null，让 axios 跳过该 header，
+     // 浏览器在发送 FormData 时会自动补充 multipart/form-data; boundary=...
+     'Content-Type': null
+   }
  })
```

#### 文件 2：`calcite-web/src/components/center/NoteEditor.vue`

将 `noteId` 通过 `uploadFile` 的第二个参数传入，而不是在组件层手动操作 `FormData`，保证接口职责单一、行为可预期。

```diff
-       const formData = new FormData()
-       formData.append('file', file)
-
-       // 关联当前笔记ID（如果存在）
-       console.log('onUploadImg: props.note =', props.note)
-       console.log('onUploadImg: props.note?.id =', props.note?.id)
-       if (props.note?.id) {
-         // 确保 note_id 是字符串（FormData 只能存字符串）
-         formData.append('note_id', String(props.note.id))
-         console.log('onUploadImg: appended note_id =', props.note.id)
-       } else {
-         console.log('onUploadImg: no note_id, file will be unlinked to any note')
-       }
-
        // 1. 上传文件，获取 file_id
-       const uploadRes = await uploadFile(formData)
+       const formData = new FormData()
+       formData.append('file', file)
+
+       // 关联当前笔记ID（如果存在）
+       const noteId = props.note?.id || null
+
+       // 1. 上传文件，获取 file_id
+       const uploadRes = await uploadFile(formData, noteId)
```

### 验证方式

- 在编辑器中上传图片，检查 Network 面板中 `/api/file/upload` 请求的 `Content-Type` 是否包含完整的 `boundary` 参数。
- 上传成功后，调用 `/api/file/list?note_id=<当前笔记ID>`，应能返回刚上传的文件记录。

## Bug-2：笔记列表界面无法查看已上传文件列表

### 根因分析

`FileList.vue` 组件设计了两种显示模式：
- **当前笔记文件**（`files`，即 `noteFiles`）
- **所有文件**（`allFiles`，即用户全部上传文件）

但数据在从 `Home.vue` → `RightSidebar.vue` → `FileList.vue` 的传递链路中存在 **多处断点**：

1. **`Home.vue` 未向 `RightSidebar` 传入 `:all-files="allFiles"`**
2. **`RightSidebar.vue` 未将 `allFiles` 继续传给 `FileList`**
3. **`RightSidebar.vue` 未转发 `FileList` 的 `@view-mode-change` 事件**
4. **`Home.vue` 中 `@file-refresh` 绑定的是 `fetchNoteFiles` 而不是 `handleFileRefresh`**，导致 `FileList` 点击刷新时传入的 `viewMode`（`'all'` / `'note'`）参数丢失，始终只刷新当前笔记文件。
5. **`Home.vue` 未监听 `@view-mode-change` 事件**，导致用户点击“查看全部”时不会触发 `fetchAllUserFiles()` 预加载数据。

### 修复内容

#### 文件 3：`calcite-web/src/views/Home.vue`

补全 `RightSidebar` 的 props 与事件绑定：

```diff
  <RightSidebar
    :all-tags="allTags"
    :note-tags="noteTags"
    :editing-note="editingNote"
    :is-tag-bound="isTagBound"
    :files="noteFiles"
+   :all-files="allFiles"
    :files-loading="filesLoading"
    @create-tag="handleCreateTag"
    @tag-action="handleTagAction"
    @tag-click="handleTagClick"
    @file-delete="handleDeleteFile"
-   @file-refresh="fetchNoteFiles"
+   @file-refresh="handleFileRefresh"
+   @view-mode-change="handleViewModeChange"
  />
```

#### 文件 4：`calcite-web/src/components/sidebar/RightSidebar.vue`

将 `allFiles` 和事件完整透传给 `FileList`：

```diff
  <FileList
    :files="files"
+   :all-files="allFiles"
    :loading="filesLoading"
    @delete="$emit('file-delete', $event)"
-   @refresh="$emit('file-refresh')"
+   @refresh="$emit('file-refresh', $event)"
+   @view-mode-change="$emit('view-mode-change', $event)"
  />
```

### 修复后行为

| 场景 | 预期行为 |
|------|----------|
| 笔记编辑页 → 文件面板 | 显示与当前 `note_id` 绑定的文件列表 |
| 笔记编辑页 → 文件面板 → 点击“查看全部” | 显示该用户的全部上传文件 |
| 笔记列表页 → 文件面板 | 由于无当前笔记，默认显示空；点击“查看全部”后正确显示全部文件 |
| 点击“刷新列表” | 根据当前模式（当前笔记 / 全部）刷新对应列表 |

## 附：API 对照

根据 `./docs/api.md` 文件管理章节：

- `POST /api/file/upload`：请求体为 `multipart/form-data`，字段包含 `file`（必填）和 `note_id`（可选）。
- `GET /api/file/list`：支持 `user_id`、`note_id`、`status` 查询参数；不传 `note_id` 时返回当前用户全部文件。

修复后的前端实现已完全对齐上述接口规范。

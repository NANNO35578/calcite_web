# Hotfix 文档 — 2026-04-13 文件前端过滤与标签管理修复

## 概述

本次修复解决了 `./calcite-web` 项目中与文件列表和标签管理相关的 **2 个前端 Bug**：

1. **Bug-1**：用户要求查询已上传文件时不再向后端传递 `note_id` 参数，改为在前端根据当前笔记 ID 过滤显示。
2. **Bug-2**：标签管理页面（"所有标签"列表）无法删除或重命名标签。后端已提供 `tag/update` 和 `tag/delete` API，但前端缺少对应的交互入口和事件处理。

---

## Bug-1：文件列表改为前端过滤

### 需求说明

- 原逻辑在打开笔记编辑页时会额外调用 `getFileList({ note_id: ... })` 获取当前笔记的文件列表，单独维护一个 `noteFiles` 状态。
- 现要求统一只获取用户全部文件（`allFiles`），在 `FileList.vue` 组件内部根据传入的当前笔记 ID 进行前端过滤。

### 修复内容

#### 文件 1：`calcite-web/src/components/sidebar/FileList.vue`

- 增加 `noteId` prop（`Number`，默认 `null`）。
- 修改 `displayFiles` computed 中 `"current note"` 状态的过滤条件：

```diff
  if (statusFilter.value === 'current note') {
-   return fileList.filter(f => f.note_id && f.note_id !== 0)
+   return fileList.filter(f => f.note_id === props.noteId)
  }
```

#### 文件 2：`calcite-web/src/components/sidebar/RightSidebar.vue`

- 向 `FileList` 组件传入当前笔记 ID：

```diff
  <FileList
    :all-files="allFiles"
+   :note-id="editingNote?.id"
    :loading="filesLoading"
    ...
  />
```

#### 文件 3：`calcite-web/src/views/Home.vue`

- **移除** `noteFiles` ref 和 `fetchNoteFiles` 函数，当前笔记的文件显示完全由 `FileList.vue` 基于 `allFiles` 前端计算。
- 清理所有引用 `noteFiles` 和 `fetchNoteFiles` 的代码：
  - `openNoteEditor()` 中移除 `fetchNoteFiles()` 调用
  - `closeEditor()` / `handleDeleteNote()` / `handleSaveNote()` 中移除 `noteFiles.value = []`
  - `handleDeleteFile()` / `handleFileUploaded()` 中移除 `fetchNoteFiles()` 调用
  - `handleFileRefresh()` 简化为只调用 `fetchAllUserFiles()`
  - `handleFolderClick()` 中移除 `noteFiles.value = []`

## Bug-2：标签管理支持删除与重命名

### 需求说明

- 后端 API 文档（`./docs/api.md` Line 428~588）提供了：
  - `POST /api/tag/update` —— 重命名标签（参数：`tag_id`, `name`）
  - `POST /api/tag/delete` —— 删除标签（参数：`tag_id`）
- 当前标签管理页仅支持行内创建和点击绑定/解绑，缺少删除和重命名功能。
- 用户提示在 `tag-list-header` 中添加编辑按钮来管理已创建的标签。

### 修复内容

#### 文件 4：`calcite-web/src/components/sidebar/TagList.vue`

- 增加 `editable` prop（`Boolean`，默认 `false`），用于控制是否显示管理功能。
- 在 `.tag-list-header` 右侧增加"管理标签"按钮（`EditPen` 图标），点击后切换 `manageMode`：
  - 非管理模式下，标签维持原有行为（点击绑定/解绑、创建新标签）。
  - **管理模式下**：
    - 所有标签显示 `closable`（可点击 `×` 删除）。
    - 每个标签右侧显示编辑图标按钮，点击后该标签变为输入框，输入新名称并按回车或失焦后确认重命名。

新增状态与方法：
```js
const manageMode = ref(false)
const editingTag = ref(null)
const editingName = ref('')

const startEdit = (tag) => { ... }
const confirmEdit = () => { emit('edit', { tag: editingTag.value, newName: editingName.value }) }
```

新增 `edit` emit，事件参数为 `{ tag, newName }`。

#### 文件 5：`calcite-web/src/components/sidebar/RightSidebar.vue`

- "所有标签"的 `TagList` 启用 `editable`，并绑定编辑与删除事件：

```diff
  <TagList
    :tags="allTags"
    title="所有标签"
    empty-text="暂无标签"
    :loading="tagsLoading"
    :closable="false"
+   :editable="true"
    @click="handleTagClick"
    @create="handleCreateTagInline"
+   @delete="handleTagDeleteAll"
+   @edit="handleTagEdit"
  />
```

- 新增事件转发：
  - `handleTagEdit(payload)` → `emit('tag-edit', payload)`
  - `handleTagDeleteAll(tag)` → `emit('tag-delete-all', tag)`
- `defineEmits` 中补充 `'tag-edit'` 和 `'tag-delete-all'`。

#### 文件 6：`calcite-web/src/views/Home.vue`

- `RightSidebar` 组件绑定中新增事件监听：

```diff
  <RightSidebar
    ...
    @tag-delete="handleTagDelete"
+   @tag-edit="handleTagEdit"
+   @tag-delete-all="handleTagDeleteAll"
    @file-delete="handleDeleteFile"
    ...
  />
```

- 新增标签重命名处理函数 `handleTagEdit`：

```js
const handleTagEdit = async ({ tag, newName }) => {
  try {
    await updateTag({ tag_id: tag.id, name: newName })
    ElMessage.success('标签修改成功')
    fetchAllTags()
    if (editingNote.value) {
      await fetchNoteTags()
    }
  } catch (error) {
    console.error('修改标签失败:', error)
  }
}
```

- 新增标签删除处理函数 `handleTagDeleteAll`：

```js
const handleTagDeleteAll = async (tag) => {
  ElMessageBox.confirm(`确定要删除标签 "${tag.name}" 吗？删除后所有关联的笔记将失去该标签。`, '提示', {
    confirmButtonText: '确定',
    cancelButtonText: '取消',
    type: 'warning'
  }).then(async () => {
    try {
      await deleteTagApi({ tag_id: tag.id })
      ElMessage.success('标签删除成功')
      fetchAllTags()
      if (editingNote.value) {
        await fetchNoteTags()
      }
    } catch (error) {
      console.error('删除标签失败:', error)
    }
  }).catch(() => { })
}
```

> 注：`handleTagDelete` 仍负责"当前笔记标签"的**解绑**操作（调用 `bindTag`），`handleTagDeleteAll` 负责"所有标签"的**真正删除**操作（调用 `deleteTagApi`）。

## 附：API 对照

根据 `./docs/api.md`：

- `POST /api/tag/update`：`{ tag_id, name }` → 更新标签名称
- `POST /api/tag/delete`：`{ tag_id }` → 删除标签（同时清除与所有笔记的关联）

前端实现已完全对齐上述接口规范。
